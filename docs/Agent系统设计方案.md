# Storyteller Agent 系统设计方案

**文档版本**: v1.0  
**整理日期**: 2026-03-14  
**文档定位**: Storyteller 互动式小说生成系统总体设计方案

---

## 1. 系统目标

Storyteller 的目标不是做一个单次续写器，而是做一个可长期运行、逻辑可控、支持多故事与多分支的互动式小说 Agent 系统。

系统应具备以下核心能力：

- 支持多个独立故事并行存在
- 每个故事下支持多个会话分支
- 每个分支都具备完整运行状态
- 围绕玩家输入按需调用 skill
- 支持低、中、高、超高四档推理强度
- 支持长程剧情一致性维护
- 支持结构化变量追踪
- 支持剧本全局推进与副本动态触发
- 支持多题材泛化适配

---

## 2. 技术基座

当前系统的基础技术栈应以以下形态设计：

- 后端：`FastAPI`
- ORM：`SQLAlchemy`
- 主数据库：`PostgreSQL`
- 向量检索：多知识域分层检索 + 统一 embedding 索引
- 模型接入：OpenAI 兼容接口与可扩展 LLM 配置体系

PostgreSQL 作为主数据库后，系统设计应默认具备以下能力预期：

- 更稳定的并发与事务能力
- 更适合复杂关系建模
- 更适合多故事、多分支、多事件账本的结构化存储
- 后续更容易扩展 JSONB、全文检索、分区表、索引优化

因此，本方案不再以 SQLite 约束为前提，而以 PostgreSQL 为正式运行底座进行设计。

---

## 3. 核心对象模型

系统建议围绕五个一级对象展开：

- `Story`
- `SessionState`
- `StorySegment`
- `EventLedger`
- `VariableState`

这五者的关系应为：

- `Story` 表示一个独立故事
- `SessionState` 表示该故事下的一个可运行分支
- `StorySegment` 表示该分支上的剧情片段
- `EventLedger` 表示从剧情中沉淀出的结构化事实
- `VariableState` 表示当前分支在当前时刻的世界数值真相快照

---

## 4. 多故事结构设计

### 4.1 Story 的定位

`Story` 应作为高于 `SessionState` 的一级对象存在。

它代表一个完整的故事工程，而不是一次运行分支。

一个 Story 应至少包含：

- 故事 ID
- 故事标题
- 故事简介
- 题材标签
- 当前采用的世界观范围
- 默认剧本
- 可用角色集合
- 可用世界书集合
- 可用预设集合
- 可用副本模板集合
- 创建者 / 所属用户

### 4.2 为什么必须有 Story 层

如果没有 Story 层，系统会天然退化为“单故事单运行”的模型。  
但你明确要求的是：

- 不是单一故事
- 可以有其他故事

因此系统结构必须是：

- 用户
- 故事
- 故事分支会话
- 剧情片段

而不是：

- 用户
- 单一会话

### 4.3 Story 的职责

Story 主要负责定义该故事的可用边界与资源范围，包括：

- 使用哪些角色
- 使用哪些世界书
- 使用哪套剧本主线
- 默认采用哪些预设
- 允许哪些副本模板参与
- 属于什么题材

Story 本身不直接承载运行时分支状态。  
运行时状态由 `SessionState` 承担。

---

## 5. SessionState 设计

### 5.1 SessionState 的定位

`SessionState` 应被定义为：

> 某个 Story 下的一个可运行分支状态对象

它既不是简单的前端会话，也不是只记录当前副本节点的轻量存档。  
它是一个完整的故事分支快照。

### 5.2 SessionState 应包含的内容

每个 SessionState 应至少绑定：

- 所属 `story_id`
- 根分支 ID
- 父分支 ID
- 分叉来源剧情片段
- 分支名称
- 分支类型
- 分支状态
- 当前主角 / 玩家角色
- 当前角色作用域
- 当前世界书作用域
- 当前剧本
- 当前副本实例
- 当前激活预设
- 当前激活 LLM 配置
- 当前激活模型
- 当前推理强度
- 当前短期摘要
- 当前中期摘要
- 当前关键变量快照索引
- 当前最近事件游标

### 5.3 SessionState 与全局设置的关系

全局设置中的预设、LLM 配置、默认规则只是候选池。

真正参与一次剧情生成的，是 SessionState 当前绑定的运行配置。

也就是说：

- 全局设置 = 待选资源
- SessionState = 当前生效配置

### 5.4 多分支机制

每个 Story 下应支持多个 SessionState。

建议支持的分支类型包括：

- 主线分支
- 手动分支
- 自动检查点分支
- 死亡结局分支
- 实验分支

### 5.5 死亡回档机制

当玩家角色死亡时，系统应：

1. 将当前 SessionState 标记为死亡结局
2. 写入死亡事件到账本
3. 保留完整分支状态
4. 允许从该 Story 下其他历史分支继续
5. 允许从当前分支历史节点重新分叉

---

## 6. 剧本与副本设计

### 6.1 剧本 Script 的定位

剧本是故事的全局发展骨架。

它的特点是：

- 持续存在
- 强约束
- 决定主线与世界大事件走向
- 即使玩家偏离，也会继续发展

剧本的职责包括：

- 定义全局阶段
- 定义核心节点
- 定义重要事件的推进条件
- 定义不可提前发生的内容
- 定义世界发展节奏

### 6.2 副本 Dungeon / Scenario 的定位

副本是动态触发的局部参与式剧情单元。

它的特点是：

- 由系统根据上下文决定是否触发
- 尽量短
- 强调局部体验
- 可以服务于支线、事件、挑战、局部冲突
- 可以有多种结局

副本例子：

- 探险
- 案件
- 博弈
- 某角色私人事件
- 任务
- 冲突
- 其他

### 6.3 剧本与副本的关系

建议关系如下：

- 剧本负责全局推进
- 副本负责局部体验
- 副本可作为剧本节点的局部展开
- 副本也可独立触发，但结果要回流到事件账本与剧本状态

---

## 7. 预设系统设计

### 7.1 预设的职责

预设主要影响：

- 文风
- 人物行动细节
- 叙事风格
- 思考方式
- 输出偏好
- 情绪控制
- 规划倾向

### 7.2 预设的层次

建议将预设拆为以下几层：

- 文风层
- 角色表现层
- 思考层
- 规划层
- 输出层

### 7.3 预设的运行方式

预设不应只在全局设置里作为一个被激活的对象存在。  
预设必须进入 SessionState 绑定。

即：

- 全局预设列表是资源池
- Story 可以定义默认预设范围
- SessionState 决定当前分支使用哪一套预设

这样不同故事、不同分支就可以拥有不同写法和不同思考风格。

---

## 8. LLM 配置设计

### 8.1 LLM 配置的职责

LLM 配置决定：

- 调用哪个服务端点
- 使用哪个模型
- 是否流式输出
- 默认温度与推理策略

### 8.2 运行绑定原则

LLM 配置也必须与 SessionState 绑定。

不应依赖“全局当前激活的一个模型配置”去决定所有故事分支的生成行为。

建议结构：

- 全局 LLM 配置列表
- Story 可定义默认模型范围
- SessionState 决定当前实际使用的配置与模型

---

## 9. 检索底座设计

### 9.1 设计目标

检索系统不能只围绕世界书构建。  
应构建一个多知识域、可路由、可统一重排的检索底座。

### 9.2 建议的知识域

建议至少划分以下知识域：

- 历史剧情域
- 摘要记忆域
- 世界书域
- 角色域
- 剧本域
- 副本域
- 事件账本域
- 变量快照域

### 9.3 是否应合并为单一知识库

建议采用：

- 逻辑上分域
- 物理上统一 chunk 规范与 embedding 索引
- 检索时按路由策略分域召回，再统一重排

这样比“完全混在一起检索”更合理，因为：

- 不同知识域的用途不同
- 时间性与稳定性不同
- 运行态知识与静态设定应区分处理

### 9.4 PostgreSQL 下的检索实现方向

基于 PostgreSQL，建议未来支持：

- 知识 chunk 表
- embedding 表
- JSONB 元数据
- 必要时引入向量扩展或外部向量索引

短期不强制要求一步到位，但设计上应以统一知识切块体系为终局目标。

---

## 10. 推理强度设计

推理强度由玩家主动选择，因此所有强度都必须支持长期持续生成。

### 10.1 推理强度影响范围

推理强度主要影响：

- 首轮检索的知识域数量
- 上下文剧情长度
- 是否允许中途再检索
- 再检索次数
- 变量分析细度
- 校验强度
- 回写严格度

### 10.2 低强度

适合：

- 日常推进
- 普通互动
- 低成本持续游玩

特征：

- 最小必要检索
- 上下文动态裁剪
- 轻量变量分析
- 默认不频繁中途再检索

### 10.3 中强度

适合：

- 一般剧情推进
- 多角色互动
- 有局部状态变化的场景

特征：

- 更多知识域召回
- 启用基础重排
- 允许一次补检索
- 启用更稳定的规划

### 10.4 高强度

适合：

- 战斗
- 谋略
- 剧情转折
- 关键事件推进

特征：

- 深度变量分析
- 剧本约束强检查
- 多源召回
- 允许生成中补检索

### 10.5 超高强度

适合：

- 主线大节点
- 生死结局
- 世界级事件
- 高一致性章节

特征：

- 多阶段规划
- 多轮补检索
- 多轮校验
- 严格事件回写

---

## 11. 上下文与动态检索

### 11.1 上下文不是固定大包

系统不应先拼出一个固定上下文包，再机械生成完整正文。  
更合理的设计应是：

- 先读取 SessionState 绑定
- 做基础计划
- 做首轮必要检索
- 在分析和生成过程中按需决定是否追加检索

### 11.2 上下文剧情长度动态化

每轮都需要带入部分上下文剧情，但长度应动态调整。

影响因素包括：

- 推理强度
- 当前输入长度
- 最近事件密度
- 当前剧本节点复杂度
- 当前模型上下文预算

### 11.3 中途再检索的原则

中途再检索不是默认必须动作，而是受成本控制的可选能力。

建议：

- 低强度：尽量避免，仅在信息严重缺失时触发
- 中强度：允许一次补检索
- 高强度：允许多次补检索
- 超高强度：允许按规划多轮补检索

---

## 12. 动态执行流程

### 阶段 0：接收请求

输入：

- `story_id`
- `session_id`
- 玩家输入
- 推理强度

读取：

- SessionState 当前运行绑定

### 阶段 1：任务识别

识别本轮是：

- 对话
- 交易
- 探索
- 战斗
- 推理
- 主线推进
- 副本触发
- 结局风险

### 阶段 2：基础计划

形成本轮首批执行计划，决定：

- 先用哪些 skill
- 先检索哪些知识域

### 阶段 3：首轮检索

根据计划执行首轮必要检索。

### 阶段 4：变量与约束分析

分析：

- 当前关键角色
- 当前关键事件
- 当前剧本推进要求
- 当前副本状态或触发可能

### 阶段 5：剧情规划

得出本轮叙事目标、禁止事项、必要事件和可选推进方向。

### 阶段 6：生成前补检索判断

若信息不足，则根据推理强度决定是否追加检索。

### 阶段 7：正文生成

在预设、约束、变量结果和检索结果的基础上生成输出。

### 阶段 8：生成中补检索

仅在允许的推理强度下开放。  
当规划或生成过程中发现信息不足时，触发补检索。

### 阶段 9：结果校验

校验：

- 输出格式
- 事实一致性
- 剧本约束
- 变量冲突

### 阶段 10：事件回写

提取关键事件，更新：

- 事件账本
- 变量状态
- SessionState
- 剧本推进状态
- 副本状态
- 摘要记忆

## 13. 输出格式设计

当前前端已依赖 [output_format_constraint.txt](Storyteller/backend/core/output_format_constraint.txt) 中的四个顶层标签进行渲染，因此短期内应保持兼容。

当前四个顶层区域为：

- `<思考过程>`
- `<正文部分>`
- `<内容总结>`
- `<行动选项>`

### 13.1 短期建议

不改变顶层标签，只收紧内部语义。

建议形式：

```xml
<思考过程>
本轮目标: ...
关键依据: ...
约束提醒: ...
候选变化: ...
</思考过程>

<正文部分>
正式剧情正文。
</正文部分>

<内容总结>
事件: ...
状态变化: ...
当前局势: ...
</内容总结>

<行动选项>
1. ...
2. ...
3. ...
</行动选项>
```

### 13.2 长期建议

长期可逐步分离：

- 内部推理输出
- 玩家可见输出

但在当前前端协议下，不建议立即拆分。

## 14. 变量与事件账本设计定位

系统长期稳定运行的核心在于两层结构化中枢：

- 事件账本
- 变量状态

剧情文本只能作为原始叙事来源，不应直接作为系统真相。

建议三层关系：

- 剧情文本：原始叙事
- 事件账本：结构化事实
- 变量状态：当前真相快照

事件账本负责记录（需要适配多种类型的故事）：

- 死亡
- 突破或阶段跃迁
- 关系变化
- 物品获得与消耗
- 地点变化
- 剧本节点推进
- 副本进入与结局
- 世界级事件（背景性事件）
- 分支创建与死亡结局

---

## 15. 泛化性设计

系统必须服务多题材，而不是内建某一种小说语义。

### 15.1 必须适配的题材范围

- 修仙
- 玄幻
- 仙侠
- 都市
- 架空
- 历史
- 西幻
- 科幻
- 悬疑

### 15.2 泛化设计原则

- 使用中性实体类型
- 使用可扩展属性容器
- 使用题材标签与 schema
- 使用预设与世界书实现风格适配
- 使用抽象变量类别，不写死题材专属字段

例如：

- 修仙中的“突破”
- 都市中的“晋升”
- 历史中的“封爵”
- 西幻中的“升阶”

都应统一抽象为：

- 阶段跃迁事件

---

## 16. 低强度优先的开发策略

虽然系统最终支持四档推理强度，但开发顺序可以从低强度开始。

前提是：

- 整体架构一次设计清楚
- 低强度只是同一系统的轻量执行模式
- 不要为低强度单独做一套无法扩展的特殊逻辑

因此建议第一阶段先完成：

- Story 层与 SessionState 层结构落地
- Session 级绑定机制
- 多分支骨架
- 低强度基础执行链路
- 事件账本骨架
- 变量状态骨架
- 剧本 / 副本双轨运行骨架

在此基础上再逐步增加中高强度特性。

---

## 17. 系统结论

Storyteller 的 Agent 系统应被设计成一个：

- 基于 PostgreSQL 的多故事叙事系统
- 以 Story 为上层资源边界
- 以 SessionState 为分支运行核心
- 以剧本控制全局发展
- 以副本提供动态局部体验
- 以多知识域检索为事实底座
- 以事件账本和变量状态维护长期一致性
- 以推理强度控制成本与深度
- 能在低强度下长期稳定运行
- 并自然扩展到更高推理强度的精密互动小说 Agent
